# Nix Build Documentation

This document covers building `wire-cli` with Nix and managing dependencies for reproducible builds.

## Overview

This project uses [buildGradleApplication](https://github.com/raphiz/buildGradleApplication) from nixpkgs for reproducible Gradle builds. This requires a `verification-metadata.xml` file that contains cryptographic checksums of all dependencies.

## Prerequisites

- [Nix](https://nixos.org/download.html) with flakes enabled
- The project uses Gradle 9.1.0 and JDK 17

## Building with Nix

```bash
# Build the package
nix build

# Run the built application
./result/bin/wire-cli --help

# Enter development shell
nix develop
```

## Updating Dependencies

### When to Update verification-metadata.xml

You need to regenerate `verification-metadata.xml` when:

1. **Adding new dependencies** to `build.gradle.kts`
2. **Updating dependency versions** in `build.gradle.kts` or `gradle/libs.versions.toml`
3. **Updating Gradle version** or wrapper properties
4. **Updating plugins** in `settings.gradle.kts` or `build.gradle.kts`
5. **After modifying** the included build (`vendor/kalium`)

### How to Regenerate verification-metadata.xml

#### Option 1: Quick Refresh (for minor changes)

Use this when adding a few new dependencies without version changes:

```bash
nix develop --command ./gradlew --write-verification-metadata sha256 dependencies
```

#### Option 2: Full Refresh (recommended for version updates)

Use this when updating dependency versions or when the quick refresh fails:

```bash
nix develop --command ./gradlew --refresh-dependencies --write-verification-metadata sha256 --write-locks build installDist
```

This command:
- `--refresh-dependencies`: Forces Gradle to re-download dependencies
- `--write-verification-metadata sha256`: Generates checksums for all artifacts
- `--write-locks`: Updates lock files for dependency constraints
- `build installDist`: Builds the project to ensure all dependencies are resolved

#### Option 3: Using buildGradleApplication Tool

The buildGradleApplication project provides a dedicated tool:

```bash
nix run github:raphiz/buildGradleApplication#update-verification-metadata
```

## Troubleshooting

### Missing Artifacts

If you see errors about missing artifacts (e.g., `detekt-rules`):

1. Check if the dependency exists in the declared repositories
2. Some artifacts may be hosted on custom repositories (Ivy, GitHub raw) that don't follow Maven patterns
3. For development-only tools (like detekt), consider removing the dependency for Nix builds if not needed

**Example - detekt-rules issue:**
The `com.wire:detekt-rules` is hosted on a custom Ivy repository. Since detekt is only for linting and Nix builds don't run detekt, this dependency is patched out in `flake.nix`.

### Repository Configuration Issues

If dependencies cannot be found:

1. Verify the repository URL is correct in `repositories` in `flake.nix`
2. Check if the repository requires authentication
3. Ensure the repository follows Maven repository patterns (required by buildGradleApplication)

**Current repositories configured:**
- `https://repo1.maven.org/maven2/` (Maven Central)
- `https://dl.google.com/android/maven2/` (Google Android)
- `https://plugins.gradle.org/m2/` (Gradle Plugin Portal)

### Composite Build (Kalium) Dependencies

This project includes `vendor/kalium` as a composite build. When updating kalium:

1. Update the kalium input in `flake.nix`:
   ```bash
   nix flake update kalium
   ```

2. Regenerate verification metadata:
   ```bash
   nix develop --command ./gradlew --refresh-dependencies --write-verification-metadata sha256 --write-locks build installDist
   ```

3. The `flake.nix` patches kalium's build to work with Nix:
   - Adds `FAIL_ON_PROJECT_REPOS` for centralized repository management
   - Removes Ivy repository references (not supported by buildGradleApplication)
   - Patches protobuf plugin to use Nix's protoc

### Gradle Version Mismatch

If you see errors about Gradle version:

1. Check `gradle/wrapper/gradle-wrapper.properties` for the current version
2. Update `gradleVersion` in `flake.nix` to match
3. Regenerate verification metadata

### Build Fails After Verification Metadata Update

1. Ensure you've committed the updated `verification-metadata.xml`
2. Try a clean build:
   ```bash
   nix build --rebuild
   ```

## Files to Commit After Dependency Upgrade

After updating dependencies and regenerating verification metadata, commit these files:

```bash
# Required
gradle/verification-metadata.xml

# If it exists (generated by --write-locks)
gradle/verification-metadata.lock

# If you updated flake inputs
flake.lock
```

Example commit:
```bash
git add gradle/verification-metadata.xml flake.lock
git commit -m "chore: update dependencies"
```

## Development Workflow

### Local Development (without Nix)

```bash
# Build
./gradlew build

# Run
./gradlew run

# Test
./gradlew check
```

### Local Development (with Nix shell)

```bash
# Enter development environment
nix develop

# Build
gradle build

# Run
gradle run
```

### Pre-Commit Checklist

Before committing changes that affect dependencies:

1. [ ] Build succeeds: `nix build`
2. [ ] Tests pass: `./gradlew check`
3. [ ] Verification metadata is up to date (no missing artifact errors)
4. [ ] `flake.lock` is updated if inputs changed
